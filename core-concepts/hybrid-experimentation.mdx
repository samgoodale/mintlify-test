- Implement and deploy variations more easily on the server side, while tracking to third-party analytics tools can be performed more effectively using the Kameleoon engine via the application file `engine.js` (previously named `kameleoon.js`), as many third-party web analytics solutions only operate on the front end.
- Target users by using data collected client-side, such as goal conversions or custom data based on data layer variables, within segments.
- Filter and break down data in Kameleoon reports using [25+ data points](https://help.kameleoon.com/experiments-results-charts/#Breakdownnbsp) collected client-side by our front-end application engine.
- Exclude users from a feature flag if they have been exposed to a web experiment or personalization.

This article describes how to implement the optimal integration between the Kameleoon [web SDK](https://developers.kameleoon.com/feature-management-and-experimentation/overview#server-side-sdks) and the Kameleoon [front-end application file](https://developers.kameleoon.com/web-experimentation/simple-implementation#asynchronous-tag-without-anti-flicker) so that the web SDK and front-end application file communicate with each other.

Use the [Kameleoon Asynchronous tag without anti-flicker](https://developers.kameleoon.com/web-experimentation/simple-implementation#asynchronous-tag-without-anti-flicker).

<Tip>
**Tip:**

For web experimentation, you normally need to add the JavaScript installation tag either in the `<head>` tag or at the beginning of the `<body>` tag to prevent flickering. If you're using hybrid mode where flickering is not a concern, you can install the script anywhere before the closing `</body>` tag since the tag is only used for tracking purposes.
</Tip>

Hybrid experimentation is currently only compatible with the Web SDKs. While the Mobile SDKs do not yet support hybrid mode with third-party web analytics tools, data points collected on a website can still be leveraged if both a website and a mobile app are offered. Data gathered from an identified user on a website can also be used in a mobile application, provided the visitor is identified consistently across both platforms.

## Synchronize user IDs between the back-end and front-end

One of the challenges of hybrid mode is ensuring that Kameleoon can properly identify each visitor as the same individual on both the server side and client side. In Kameleoon, the `visitorCode`, which is a unique ID for every visitor, must be identical on both the server side and client side. For example, a visitor registered with the `visitorCode` of `10ctbql0zpf4rwjy` for an experiment variation in the back-end must be associated with the visitor code `10ctbql0zpf4rwjy` when a conversion is triggered in the front-end. Using the same ID ensures correct tracking of the visitor's actions, which results in correct data and statistics.

To synchronize the user IDs, the hybrid app must call the `getVisitorCode()` method prior to calling other SDK methods. Since naming conventions vary between languages, the method names vary slightly depending on the SDK, but each language has a similarly named equivalent method. Kameleoon uses the following approach:

1. On the front-end, the Kameleoon JavaScript engine randomly generates a `visitorCode` for each new visitor. The engine considers a visitor to be new if the engine can't find a previous visitor code on the user's browser (for example, in a cookie or local storage). The generation normally happens when the Kameleoon engine runs for the first time on the visitor's browser. The Kameleoon engine can also obtain the identifier from a cookie that the back-end passes to the engine. The cookie option allows setting the identifier on the server, rather than in the front-end. For example, use the cookie option to specify a custom unique visitor code for each visitor.

<Note>
**Note:**

If you [Unify session data accross subdomains](https://developers.kameleoon.com/core-concepts/unify-session-data-across-subdomains), Kameleoon looks for a **kameleoonVisitorCode** key in local storage. If the key is present, its value is used as the identifier for the visitor code. The local storage version of this key always takes precedence, even if a **kameleoonVisitorCode** cookie with a different value exists. Note that Kameleoon does not reset the JavaScript cookie if the local storage and cookie values match. For ITP purposes, it is important to leave cookies that may have been set on the server side intact.
</Note>

2. On the back-end, call the `getVisitorCode()` method or equivalent in the SDK. Call the `getVisitorCode()` method before providing a visitor identifier to any other SDK methods. The `getVisitorCode()` method takes an optional string argument to pass a custom identifier rather than rely on a randomly generated Kameleoon ID. When the method is called, Kameleoon first looks for a **kameleoonVisitorCode** cookie or query parameter associated with the current HTTP request. If found, the SDK uses the cookie or query parameter as the visitor identifier.

If no cookie or parameter is found, Kameleoon uses the `getVisitorCode()` method argument, if provided, as the identifier. If no argument is provided, Kameleoon randomly generates the ID. In either case, Kameleoon also sets the server-side **kameleoonVisitorCode** cookie with the value (via HTTP header).

By following the synchronization approach, the **visitorCode** is saved and shared between front-end and back-end. If an experiment is implemented on the first page of a visitor's journey on the website, the SDK generates the identifier and passes the identifier to the JavaScript engine. If the journey starts on a page where `getVisitorCode()` is not called (or the synchronization snippet is not run) on the back-end, then the JavaScript engine generates the identifier first and passes the identifier to the SDK. The SDK reads the identifier and then rewrites the identifier as a server-side cookie to bypass ITP restrictions.

## Linking feature experiments with front-end tracking code

Setting up experiment variations using an SDK while implementing tracking on the front-end using the JavaScript engine is often useful. The approach is common for the following reasons:

- Web analytics platforms (such as Mixpanel or Google Analytics 4) tend to be implemented on the front-end.
- A standard client-side tagging plan for Kameleoon using [web experimentation](https://www.kameleoon.com/en/platform/ab-testing-client-side) may already be in use.
- Tracking events and [data points](https://help.kameleoon.com/experiments-results-charts/#Breakdownnbsp) on the front-end side for analytics purposes with Kameleoon reporting may be more convenient.
- Some [data points](https://developers.kameleoon.com/feature-management-and-experimentation/web-sdks/js-sdk#targeting-conditions) for feature experiments are easier to collect on the front-end side.

### Sending exposure events to third-party analytics


Kameleoon provides native [integrations](https://help.kameleoon.com/category/user-manual/manage-your-integrations/) with analytics and CDP platforms. In hybrid mode, any of these integrations can be used in combination with feature experiments, without needing to write additional code.

The Kameleoon front-end engine must know whenever an experiment has taken place in the back-end. That is, when a visitor has been bucketed into an experiment. Knowing when an experiment has taken place is important to count only visitors that have actually triggered and seen the experiment. To achieve the notification, use the `getEngineTrackingCode()` method (the name varies slightly depending on the SDK), which is available in all Kameleoon server-side SDKs. The `getEngineTrackingCode()` method returns the JavaScript code to be inserted in the page to automatically send the exposure events from the front-end to the analytics solution in use. The JavaScript code can be embedded in the returned HTML page.

For example:

```html 
<!DOCTYPE html>
<html lang="en">
  <body>
    <script>
      const engineTrackingCode = \`
        window.kameleoonQueue = window.kameleoonQueue || [];
        window.kameleoonQueue.push(['Experiments.assignVariation', 123456, 7890]);
        window.kameleoonQueue.push(['Experiments.trigger', 123456, true]);
      \`;
      const script = document.createElement('script');
      script.textContent = engineTrackingCode;
      document.body.appendChild(script);
    </script>
  </body>
</html>
```

<Note>
**Note:**

[Activate](https://help.kameleoon.com/manage-third-party-analytics-integrations/) the chosen integration (for example, Google Analytics 4, Mixpanel, etc) when configuring the feature experiment. The relevant data (such as experiment ID, experiment name, variation ID, variation name) will be automatically sent to the third party platform by the Kameleoon JavaScript engine.
</Note>

### Using front-end data points and events in feature experiments

If Kameleoon is implemented in hybrid mode, goals and [data points](https://help.kameleoon.com/experiments-results-charts/#Breakdownnbsp) are automatically tracked on the front end and appear in Kameleoon reports as filtering and breakdown criteria. In addition, several [targeting conditions](https://developers.kameleoon.com/feature-management-and-experimentation/using-visit-history-in-feature-flags-and-experiments) are available in the Kameleoon SDK, ready to be used for targeting purposes.

### Managing consent in hybrid mode

When using Kameleoon in Hybrid mode and [consent is set to "Required"](https://help.kameleoon.com/consent-management-policy/), ensure that both the JavaScript Activation API and SDK methods are called upon collecting consent so that Kameleoon can collect user events from both the Kameleoon Application File and the SDK. Additionally, as Kameleoon is restricted from storing the visitorCode key in a cookie, ensure that the visitorCode key generated by the SDK for all first requests is shared with the Kameleoon engine running client-side. Read the [technical considerations](https://help.kameleoon.com/consent-management-policy/#technical_considerations) for more information.

<Note>
**Note:**

Consent information is in sync between the Kameleoon Engine (application file `engine.js`) and the JS SDK. The synchronization means that once consent is set on either the Engine or the SDK, consent is automatically set for both. The synchronization feature eliminates the need for manual consent handling and ensures that SDKs operate in compliance with user preferences.
</Note>
